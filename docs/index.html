How it holds up

Five surfaces, each load-bearing.

The whole system exists to make one flow undeniable: pay → the receipt is the pre-kickoff proof → the match settles → CLV scores the pick → it lands on the public ledger.

01

x402 pay-per-pick gate

One World Cup knockout edge per call — model probability, edge %, and a 4-rung conviction ladder, gated at 0.05 USDC over the real @injectivelabs/x402 middleware. No account, no login, no API key.

02

Receipt = pre-kickoff proof

The USDC settlement tx arrives in-band. Its on-chain block time < kickoff is the notarization, bound to the pick hash in the same ledger row — you can't backdate a bet.

03

Free CLV-scored ledger

Every settled pick is scored on Closing Line Value — losses included, and wins that lost CLV kept. The system publishes its own bad prices.

04

One-command audit

npm run audit -- --all independently re-hashes every pick (sha256), rechecks block times, and recomputes every Merkle root → "LEDGER PASSES".

05

On-chain Merkle anchor

A write-once daily Merkle root of pick hashes on LedgerAnchor.sol (Injective EVM) proves the ledger DB was never rewritten after the fact.

06

Agents are customers

The shipped worldcup-linelock Agent Skill teaches any harness to be a buyer: balance → pay → validate hash → audit — via the Injective MCP server, funded cross-chain by USDC CCTP V2.

Questions a judge would ask

Straight answers.

HTTP status 402 ("Payment Required") turned into a working payment protocol. LineLock's POST /api/edge is gated by injectivePaymentMiddleware from the real @injectivelabs/x402 package: the first request returns a 402 quote (PAYMENT-REQUIRED header), the client signs an EIP-3009 USDC authorization (PAYMENT-SIGNATURE), and the 200 response carries the settlement receipt in-band (PAYMENT-RESPONSE).

No. The 402 quote is built from config by the real middleware and needs zero wallet balance — curl -i -X POST :8402/api/edge is enough. Running npm run api -- --demo disables the gate so the same route returns the actual paid payload, self-labeled receipt: null. Only settling a real payment needs gas, and scripts/paid-call-smoke.ts does a balance preflight and refuses to fabricate a receipt when the wallet is unfunded.

One World Cup knockout edge: the model probability, the edge % versus the market, a 4-rung conviction ladder, a pick_hash commit (sha256 of the canonical pick JSON), and similar_settled — comparable picks from LineLock's own settled ledger, losses included. That's 50000 USDC units on the quote.

One command: npm run audit -- --all. It independently re-hashes every pick, checks that each receipt block time sits before kickoff (invariant I1), verifies rows == receipts (I3), and recomputes every daily Merkle root for the anchor contract (I5 — written + tested; mainnet deploy funds-gated) → "LEDGER PASSES". The audit CLI is separate from the code that writes the ledger.

Buyers can fund cross-chain from Base via USDC CCTP V2 (burn → Iris attestation → mint on Injective). The daily Merkle checkpoint is implemented in LedgerAnchor.sol for the same chain (deploy funds-gated).

Aggressively so. Demo responses label themselves receipt: null with note: "Demo/unsettled response"; the seed ledger rows are flagged is_placeholder: true with synthetic receipt hashes, labeled everywhere including STATUS.md. The ops wallet isn't funded yet, and the code path that would need funds exits rather than fake an on-chain receipt.

Anyone can post a winning screenshot. Pro syndicates track Closing Line Value: did you get a better price than the market's closing line? Positive CLV across a sample = real edge, independent of win/loss variance. LineLock computes clv_pct = entry_odds / closing_odds − 1 and deliberately keeps wins with negative CLV and losses with positive CLV — that divergence is the whole point.
